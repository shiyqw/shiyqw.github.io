SUMMARY. 

We are going to implement a lock-free concurrent Hashmap in C++. Some special modification could be using linear hashing to scale, using different data structures like cuckoo hash.

BACKGROUND.

HashMap is a really widely used data structures almost in every programming language. It has O(1) lookup, insert, modify and delete operations. Different programming languages have different interfaces for HashMap, such as unordered_map in C++ STL, HashMap in Java and dict in Python.

Thread safety is an important property in HashMap implementation. Different programming languages have different support of thread safety in HashMap implementation. C++ uses std::unordered_map as its HashMap. As described in STL documentation, STL containers are designed so that you are guaranteed to be able to have multiple threads reading at the same time but just one thread writing at the same time. Having multiple threads writing is not allowed so that user should do synchronization in caller codes. Java’s HashMap has the same thread safety as C++ while Java also provide another implementation called ConcurrentHashMap which supports multiple reads. The problem of Java’s HashMap is the performance. Since it is not a lock-free implementation, the performance will be influenced by large amount of blockings.

Performance may vary greatly while using different data structure to implement HashMap. Recently Cuckoo HashMap has the better concurrent performance rather than simple structures.

There is a project in last year comparing different structures of HashMap on concurrent performance. Their projects only consider lock-free reads. It can also be a good reference.

THE CHALLENGE. 

The main challenge is not about the difficulty to make something parallelize, but to support concurrent access to the Hashmap data structure. And since the locking version could be trivial, we are focusing on lock-free version, and here are some difficulties:

  1. Implement a correct lock-free version of concurrent hashmap.
  2. Debugging and finding out the edge cases to test the correctness is also challenging
  3. At this point we are not sure whether lock-free version could be more efficient than locking version. More design thinkings and improvements would be needed.
  4. Integrating new features into the existing design.

GOALS AND DELIVERABLES. 

In general, our goal is to design and implement a lock free HashMap which supports multi thread reads and writes. The deliverables will include: (1) A complete library and interface of our HashMap, which includes create and destruct maps, get value, set value and delete items. (2) An evaluation of performance of our HashMap, C++ unordered_map (multithreaded reads only), and Java’s ConcurrentHashMap. We aim to have at least 1.5X speed up on top of Java’s ConcurrentHashMap in write intensive requests.

If we still have time we will add more functionalities such as implementations in different data structures (we will use cuckoo at first). We will also try to optimize the performance of our HashMap.
